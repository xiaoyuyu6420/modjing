# 模镜 Constitution（项目根本约束）

> 性质：项目治理根本文档。所有 spec、计划、代码决策必须服从本文件。
> 成文：2026-06-18｜由战略讨论沉淀而来
> 位置：遵循 Spec-Kit 约定 `.specify/memory/constitution.md`

---

## 0. 一句话定位

> **做"中转站哨兵"，不做"又一个 hvoy"——监控暴毙、告警价格、自建探针，先让用户每天回来，再扩展成评测平台。**

---

## 1. 项目立场

- **做裁判，不做运动员。** 只做评测与信息聚合，不提供 API 代理服务。
- **中立先于变现。** 不接受广告排名、不卖排名位、不收中转站"保护费"。
- **聚焦先于大而全。** 把一个功能做到极致，优于把所有功能都做到平庸。
- **工具先于展示。** 先做让用户每天回来的工具（监控/告警），再做展示（排行榜）。

---

## 2. 合规约束（法律底线，不可违反）

1. **不开中转站。** 自营 API 代理有 ICP 经营许可、跨境数据合规风险，且与"裁判"立场冲突。法律风险 + 定位冲突 = 双重死亡。
2. **保留 MIT 版权声明。** 源自 llm-verify（MIT, Copyright 2026 Mintesnot Teshome）的 prompt，必须在代码/页面保留版权与许可声明。
3. **移除无许可证代码引用。** APIVerifier 无 LICENSE，默认 All Rights Reserved，不引用其代码。
4. **数据抓取仅限公开端点。** 抓取 hvoy 公开端点法律风险低，但必须维护多数据源 + 手动底线数据，避免单一来源断供。
5. **增量同步不删现有数据、不覆盖后台手动标注。**

---

## 3. 不做什么（明确禁止）

- ❌ 自己开中转站
- ❌ 抄 hvoy 积分机制（那是广告的变体，会侵蚀中立性）
- ❌ 讨好中转站运营商换取信息（换来的都是广告，不是数据）
- ❌ 跟 hvoy/lmspeed 比"谁更全"——比"谁更聚焦"

---

## 4. 差异化方向（核心切入）

hvoy 与 lmspeed 都不做的三件事：

| 方向 | 为什么是我们的 |
|------|--------------|
| 站点暴毙预警 | 中转站最大痛点是"突然不可用"，没人做提前预警 |
| 价格变动告警 | 用户需要追踪，不是一次性查看；排行榜只给"此刻"价格 |
| 用户自建掺水探针 | 输入自己的 key 自己测；hvoy 不会开放（绕过其商业模式），lmspeed 不测掺水 |

---

## 5. 产品演进路径

反 hvoy 的顺序——先工具粘性，后展示：

1. 暴毙监控（1-2 周）→ 验证痛点
2. 价格告警（1-2 月）→ 用户每天回来
3. 用户自建探针（2-3 月）→ 检测工具
4. 排行榜（3-6 月）→ 有了信任才是评测平台

---

## 6. 变现路径

- 主：企业合规咨询（裁判 + 顾问）
- 辅：价格监控订阅、高级探针报告
- 不做：广告、中转站入驻、排名售卖

---

## 7. 风险与待验证假设

待验证（决定路径是否成立）：
1. 暴毙监控是真痛点吗？（社区验证）
2. 运营者愿意为上游监控付费吗？（访谈 3 个运营者）
3. 价格告警有人订阅吗？（最小版看转化）

冷启动：用认识的运营者做种子用户（**不换免费额度，换反馈**）。C 端（监控我用的站）与运营者端（监控我的上游）共用同一套"状态监控 + 通知系统"。

退路：若路径全部不成立，技术能力（Next.js + 数据库 + 探针）换赛道复用；评测平台定位可放弃，技术能力不浪费。
